WALL-WM Data Flow: from cameras & words to motor numbers

Every stage: what tensor goes in, what operation runs, what tensor comes out.

The 10-second version. Three things go IN — camera images, a language instruction, and the robot's own pose. One thing comes OUT — a table of movement numbers the motors run. The world model (imagined future video) rides along to guide the action. Everything below is just that, in detail.

vision stream language stream robot state world-model (video) action stream

Shapes below are written as [dims] with example numbers. Symbols: Nv cameras · H,W image size · T frames · h,w,C latent size/channels · D model width · L text length · K reasoning steps · Ta action steps (chunk) · A action dims per step.

0 Inputs (what the robot has right now)
multi-view image [N_v, H, W, 3]e.g. [3, 480, 640, 3] uint8 — 3 cameras, RGB pixels
instruction "put the cup in the sink"a text string
proprio state s [D_s]e.g. [14] float — arm joint angles + gripper, the robot's sense of its own body

During training there is also a target: the true future video + true action trajectory. At run-time those are what we generate.

1 Encode vision — pixels → latents (3D VAE)
raw frames [N_v, T, H, W, 3]pixels, per camera over time
3D VAE encodercompress space ÷8 & time (1+4N → 1+N frames)
video latents [N_v, T_lat, h, w, C]e.g. [3, ≤65, 60, 80, 16] float — small codes, not pixels

Why: pixels are huge and redundant. The VAE squeezes each frame into a compact grid of numbers (the "invent your own compact state" idea). All the heavy modelling happens on these latents.

2 Tokenize vision — latents → a sequence of tokens
video latents [N_v, T_lat, h, w, C]
patchify + linearcut each latent frame into p×p patches → embed to width D
+ Camera RoPE (tags which camera)
video tokens [N_v, S, D]S = T_lat·(h/p)·(w/p) tokens per view, each a D-vector

Now the video is a list of tokens (like words), ready for a Transformer. Sight-cone + tube masks act on these tokens during training.

3 Language-Guided Reasoning — the Qwen "brain" (instruction → conditioning)

This is the planner. A vision-language model — Qwen3.5-9B — reads the current views + the instruction and reasons before anything is generated. It's needed because a raw instruction ("organize the items") is too coarse for a long task; the VLM turns it into concrete, step-appropriate guidance.

multi-view observation [N_v, H, W, 3]
instruction ℓ "put the cup in the sink"
Qwen3.5-9B VLMscene grounding · event decomposition · progress tracking · replanning
VLM hidden states H_q [L, D_vlm]

The VLM does the four reasoning jobs a static instruction can't. Its output then takes one of two forms depending on the mode:

next-event caption "grasp the cup"the VLM (or a human / agent) names the next event in words
T5 text encoderwords → embeddings
text conditioning c [L, D]e.g. [~30, D] — one D-vector per token

Event mode: the reasoner emits a natural-language next-event description that conditions one variable-length event.

VLM hidden states H_q [L, D_vlm]
Staircase decoder (MoT on Qwen)reason → K continuous latent thoughts, in parallel
reasoning latents z [K, D]a compact plan, NOT words — continuous, gradient-friendly

Unified mode: the Staircase decoder turns the VLM's reasoning into K latent "thoughts" (fast, parallel) that steer fixed-length chunk generation.

Where does this conditioning go? Into the Multi-View Video DiT's cross-attention (Stage 4), alongside the image tokens — same path as any text/observation conditioning. So Language-Guided Reasoning shapes what the world model imagines; the Action DiT inherits it indirectly through the video→action coupling (S3–S4). The language sets the goal → the world model imagines the called play → the action tower runs it.
4 World model — imagine the future (Video DiT)
noisy future video tokens [N_v, S, D]
observation y (cond.) [N_v, S, D]the known current frames (inpainting channel)
text c / latents z [L, D] / [K, D]
Multi-View Video DiTflow-matching denoise, many steps
intra-view + cross-view attention
clean future video latents [N_v, T_lat, h, w, C]
per-layer video features h^V [N_v, S, D] × depththe hidden states the action tower will read

Starts from pure noise and refines it into a coherent future, guided by the observation + the instruction. Its internal layer features (h^V) are the key hand-off to the action tower.

5 Action model — turn imagination into motion (Action DiT)
noisy action tokens [T_a, D]
proprio state s [D_s] → [1, D]
video features h^V [N_v, S, D] (per layer)
Action DiTsame depth as video tower
each layer cross-attends to matched h^V (layer coupling)
action hidden states [T_a, D]

At every layer the action tower reads the world model's features for that layer — so the motion is continuously informed by the imagined future. Flow between them is one-way (video → action).

6 Action head — hidden states → real numbers
action hidden states [T_a, D]
MLP head (regressor)continuous — no vocabulary, no sampling
action trajectory [T_a, A]e.g. [16, 14] float — 16 steps × (Δpos, Δrot, gripper) per arm

This is the "numbers, not words" head. Each row is one action step the motors can run directly.

Zoom in: Spatial-Temporal Fusion (S1→S4) — how Stages 4–5 actually connect

Figure 3(c) details how the multi-view video is fused and threaded into the action tower. It's four sub-steps that repeat in every paired block (the "× depth"). The trick: where the "view" dimension sits in the tensor decides whether cameras are isolated or interacting.

S1in Video DiTIntra-view attention

Views on the batch axis → each camera attends within itself (space + time, 3D RoPE). The inherited single-camera skill, unchanged.

S2in Video DiTCross-view attention

Regroup per frame → cameras attend to each other (Camera RoPE + sight-cone/tube masks). Added via a zero-init gate, so it ramps up during training.

S3the bridgeViewConcat

Concatenate the per-view tokens along the sequence axis into one fused sequence → this becomes the keys/values.

S4in Action DiTBlock-wise coupling

The action block cross-attends to those fused video keys/values. One-way (video → action), at every block.

VIDEO DiT block ─► [ S1 intra-view ] → [ S2 cross-view ] ─► h^V (per-view features) │ [ S3 ViewConcat: fuse views → K,V ] │ (one-way) ACTION DiT block ───────────────────────────────► [ S4 cross-attend to K,V ]

Where each runs: S1 and S2 are purely inside the Multi-View Video DiT; S3 is the hand-off prep (rearranging the video tower's outputs); S4 happens in the Action DiT. So the fusion lives in the video tower, but the coupling (S3–S4) reaches into the action tower — and the whole pattern repeats at every layer.

Two axes: temporal fusion = S1 across frames; spatial fusion = S1 within a frame + S2 across cameras. "Spatial-temporal fusion" = doing both, then coupling the result into action.

7 Execute & loop
action trajectory [T_a, A]
robot controllerinverse kinematics → motor commands
robot moves → new [N_v,H,W,3]new camera view after acting
The rolling loop: execute the chunk → observe the new scene → go back to Stage 1. Event mode re-plans at each event boundary; unified mode at each fixed chunk. (Optional: a VAE decoder can turn the video latents back into watchable pixels [N_v,T,H,W,3], but that's only for visualization — the robot never needs it.)

The whole journey is a chain of format changes: pixels → latents → tokens → hidden features → numbers. No discrete "word" appears anywhere in the action path — it stays continuous end-to-end.

Three insights for the VLA task

1. It's a translation problem across formats. A VLA model is fundamentally: images + words + state → action numbers. Everything in the middle (VAE, tokens, DiT, head) is just converting between representations so a Transformer can bridge those very different formats.
2. The world model is a helper, not the output. The imagined future video (teal) is generated so its features can guide the action tower. The robot's actual output is only the action numbers. "Dream to act better."
3. Two towers, coupled layer-by-layer, one-way. Video tower imagines; action tower acts; at each depth the action tower reads the video tower's features. Keeping it one-way protects the video model's visual knowledge from being overwritten by short-horizon action shortcuts.
