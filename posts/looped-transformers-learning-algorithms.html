Looped Transformers Are Better at Learning Learning Algorithms

Paper: Looped Transformers are Better at Learning Learning Algorithms — Yang, Lee, Nowak & Papailiopoulos, ICLR 2024, arXiv:2311.12424. The empirical companion to the expressivity posts in this category: not “what can a loop express,” but “can we actually train one, and does it pay off.”

✅ In one line

A 1-layer transformer, run in a loop with the prompt re-added every iteration and trained with a sliding loss window, learns in-context learning (linear/sparse regression, decision trees, small nets) about as well as a 12-layer transformer — at ~1/12 of the parameters (0.79M vs 9.48M) — and beats it on the harder, structured tasks. Each loop iteration behaves like one step of an iterative algorithm (gradient descent), so it trades parameters for iterations.

📄 The setup & the core bet

The task: in-context learning of a data-fitting problem

Feed the model input–output pairs and one query; it must predict the query’s label without any weight updates:

prompt  P = ( x₁, f(x₁),  x₂, f(x₂),  … ,  x_k, f(x_k),  x_test )
goal       predict  f(x_test)          measured by squared error

A fresh function f is drawn per prompt (e.g. random linear f(x)=wᵀx), so the model can’t memorize f — it must learn a learning algorithm that reads the pairs and fits on the fly.

Why a loop: matrix inverse (hard) vs gradient descent (easy, repeated)

A standard transformer must solve this in one forward pass — essentially approximating the closed-form least-squares answer w = (XᵀX)⁻¹Xᵀy. The inverse is the expensive part: it’s an elimination procedure with global coupling, so representing it needs many layers/heads (that’s why the standard model wants to be deep). Attention naturally does matrix multiplication (weighted sums), but not inversion.

Gradient descent reaches the same w using only matrix multiplies, applied repeatedly:

w_{t+1} = w_t − η · Xᵀ(X w_t − y)     ← one cheap step, over and over
Same family as Newton–Schulz. Replacing a one-shot expensive linear-algebra op (inverse, SVD) with iterated matrix multiplication is exactly the Newton–Schulz trick (and what the Muon optimizer uses to avoid an SVD). Gradient descent is the first-order member; Newton is the second-order one. A loop is the natural hardware for “repeat a cheap matmul” — so 1 layer looped can stand in for 12.

🔧 Deep dive: how Newton–Schulz actually works

The callout above is the slogan; here is the machinery, because it turns out to be the cleanest possible model of “looped block replaces one-shot linear algebra” — clean enough to reason about with a theorem instead of a probe. We use the version the Muon optimizer runs on each gradient matrix.

The op being replaced is orthogonalization, not a full SVD

Muon takes a gradient/momentum matrix G and wants the nearest semi-orthogonal matrix to it. If you did a real SVD, G = U Σ Vᵀ, that nearest orthogonal matrix is Q = U Vᵀ — same singular vectors, every singular value replaced by 1. Call it the polar factor. Computing U, Σ, V is the expensive one-shot op (~n³, GPU-unfriendly, awkward gradients). Newton–Schulz produces Q using only matrix multiplies, and never forms U, Σ, or V.

“Newton–Schulz for the SVD” is really “the SVD with singular values flattened to 1.” It does not compute or approximate any individual singular vector. It computes the orthogonalization U Vᵀ.

The one fact that makes it work

Take an odd polynomial — the classic choice is p(x) = 1.5x − 0.5x³ — and apply it to a matrix in this exact shape:

X_next = 1.5·X − 0.5·X Xᵀ X

If X = U Σ Vᵀ, this expression equals U · p(Σ) · Vᵀ. Two lines show why: X Xᵀ X = (UΣVᵀ)(VΣUᵀ)(UΣVᵀ) = U Σ³ Vᵀ, so 1.5·UΣVᵀ − 0.5·UΣ³Vᵀ = U(1.5Σ − 0.5Σ³)Vᵀ. The U and V pass straight through, untouched; the polynomial acts only on the singular values, one at a time, as the scalar map σ → 1.5σ − 0.5σ³. Iterating the matrix recurrence is just iterating that scalar map on every singular value in parallel, with the singular vectors held fixed.

It moves the singular VALUES, not the vectors

U and V are exact — never approximated. Every iteration keeps the same singular vectors. The only approximation is in Σ: the values don’t land exactly on 1 (Muon stops after 5 steps, so they sit in a band near 1). NS kills the scale information and preserves the rotational structure — exactly what an optimizer wants from a noisy, ill-conditioned gradient.

Why every singular value lands on 1

Look at the scalar map f(σ) = 1.5σ − 0.5σ³ on the interval (0, 1]:

  • Fixed points: f(σ) = σ gives σ(1 − σ²) = 0 → σ = 0 or σ = 1.
  • f(1) = 1 (a value already at 1 stays), and f′(1) = 1.5(1 − σ²) = 0 at σ = 1 → quadratic convergence into 1.
  • For any σ in (0, 1), f pushes it upward toward 1.

So you first scale G so all singular values are ≤ 1 (cheaply, divide by the Frobenius norm as an upper bound on the spectral norm), then iterate. Every value climbs to 1; every vector is preserved. Result ≈ U·I·Vᵀ = U Vᵀ in a handful of matmuls.

The small value takes ~7 steps to lift — the classic polynomial is slow near zero. That is exactly the setup for Muon’s twist.

What Muon changes: retuned coefficients

Muon does not use 1.5x − 0.5x³. Keller Jordan retuned the coefficients of a quintic p(x) = a·x + b·x³ + c·x⁵ (roughly a = 3.4445, b = −4.7750, c = 2.0315) so it lifts small singular values aggressively and drives the whole spectrum into a band near 1 in about 5 iterations. It deliberately overshoots — it does not converge exactly to 1 — because an optimizer step only needs approximate orthogonalization, so speed is the right trade. Fixed count (5), same coefficients every step, all matmuls: perfectly GPU-friendly.

The loop-transformer parallel, made precise. Muon’s classic form uses the same polynomial every step = fully weight-tied loop — Spinner’s “identical operation each iteration” stance, and a fixed iteration count T = 5 just like picking T at inference. But the frontier of NS (per-step-tuned coefficients, e.g. the Polar Express line) uses a different polynomial at each iteration to converge in fewer steps — because early steps have a different job (lift the tiny singular values) than late steps (polish everything to 1). That is precisely the per-step adapter / boosting framing: iteration t gets its own coefficients because it fits the residual of steps 1…t−1 — and here you can derive the optimal per-step polynomial by minimizing worst-case distance-to-1 over the remaining spectrum. A theorem where the loop-transformer story usually only has a probe.

The training objective (Eq. 1), in words

tune θ to minimize, averaged over
   • random prompts P            → learn the algorithm, not one function
   • prefix lengths i = 0 … k     → predict well at every context size
   • loop iterations t = b−T … b  → reach a good answer AND hold it (stable fixed point)
the squared error between ŷ and the true label f(x_{i+1}).
  • b = how deep you unroll (effective depth). Too small → diverges after training; too large → slow.
  • T = loss window: only the last T iterations are scored (truncated back-prop → saves memory).
  • Scoring a window (not just the final step) is what forces a convergent fixed point. Linear regression settles on b = 20, T = 15.

🖊 Key results

1 · Matches 12 layers with 1/12 the parameters

On linear regression it tracks the 12-layer model and nearly the optimal least-squares solver — 0.79M vs 9.48M.

2 · Needs MORE iterations than the standard model has layers

A 1-layer loop needs ~20 iterations to match an 8-layer transformer → its effective depth exceeds the standard model’s; each iteration learns a different representation. (The empirical face of “effective depth = layers × iterations” from the CoT-vs-Latent post.)

3 · Learns from fewer distinct prompts

Fewer parameters → less to fit → good error after seeing fewer distinct functions. (Sample-efficiency, not data-repetition — looping is compute-depth, not extra epochs.)

4 · Probing shows genuine iterative refinement

The standard model decodes the target quantities best around layer ~10 then moves on; the loop steadily refines them every iteration — evidence of gradient-descent-style refinement, not a one-shot leap.

5 · Wins bigger on structured tasks

Comparable-or-better across sparse linear, decision trees, 2-layer nets and OpenML tabular data — clearest wins on sparse linear and decision trees.

🧠 Discussion — the ideas that made it click

Two different “weights.” θ = the transformer’s parameters, learned in training, frozen at inference. w = the regression answer, inferred fresh from each prompt, living in the activations. The loop refines w, not θ. Looping is not re-training — it’s the frozen model running its algorithm more steps. θ updates once per training batch (after the loop, via the loss); at inference θ never changes.
Two levels of learning. Outer (training): gradient descent on θ — learns how to learn. Inner (in-context): the frozen model infers the task’s answer from the prompt examples — executes that learned algorithm. This is meta-learning: training learns a learner; ICL runs it, and the loop iterations are the visible steps of the in-context learning.
When is the label used? The in-context labels y₁…y_k are inside the prompt → inputs the loop uses every step. The query label f(x_{i+1}) is used only in the training loss, to produce the one θ update — the model never sees it as input, and at inference it doesn’t exist.
It’s a fixed-point iteration (a deterministic “Markov chain”). Y_{t+1}=M(Y_t+P) depends only on the current state → memoryless, fixed rule, converges to a fixed point Y*=M(Y*+P). Precisely: a deterministic dynamical system / Deep-Equilibrium model — which is also what gradient descent is. The stochastic cousin (random transitions) is diffusion/MDM from the other post.

⚠️ The limitation: it emulates the algorithm for its training distribution

The loop reaches a stable fixed point even beyond its trained iterations — but only for the input distribution it trained on. It has an error floor and does not learn the truly universal algorithm that a real solver (least squares) applies to any inputs. This limitation hits the deep model too — both are distribution-specific; they just fail differently.

A distinctive simplicity (conservatism) inductive bias

Fewer parameters give the loop a bias toward simpler, smaller solutions — like mild regularization. The same bias helps on some shifts and hurts on others, depending on what the shift asks for:

Why the same bias flips. Simplicity bias = conservatism = “prefer small, smooth outputs.” It helps when the shift punishes overfitting (noise, ill-conditioning — you should hold back) and hurts when the shift demands extrapolation to larger inputs/outputs (scaling — you need to stretch, and a conservative model refuses to, so it undershoots).

Quick glossary

  • Noise: labels have random error — data doesn’t sit exactly on the true line. Fitting exactly = chasing the jitter (overfitting).
  • Ill-conditioning: features are nearly redundant (e.g. x₂≈x₁), so the data has almost no spread in the “difference” direction. XᵀX has a near-zero value there, and the inverse divides by it → weights blow up. Ridge adds a little before dividing → stabilizes.
Why the conservative (shrinkage) bias fixes both:
  • Noise: shrinking the weights toward small values keeps the model from chasing random jitter → less overfit.
  • Ill-conditioning: the ridge penalty adds a small amount to that near-zero value before inverting, so you no longer divide by ~0 → the wild swings are damped, the answer stabilizes.
In both cases the fix is “don’t take the data too literally / don’t produce extreme weights” — exactly what a simplicity bias does. That’s why it wins those two columns but loses on scaling, where you need large outputs.
