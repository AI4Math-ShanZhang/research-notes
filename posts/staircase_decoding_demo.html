Why the code is different

The animation is driven by these two loops. The highlighted lines are the whole story.

① Next-token (serial)

# build reasoning one token at a time state = encode(prompt) thoughts = [] for k in range(K): # K thoughts h = state for layer in range(L): # climb ALL layers h = layer(h) tok = sample(h) # discrete word thoughts.append(tok) state = state + embed(tok) # ^ next thought DEPENDS on this one # → must wait → serial → slow

② Staircase (staggered pipeline)

# shared bottom flight, built once h = encode(prompt) for layer in range(0, Nr): # bottom Nr layers h = layer(h) shared = h # reused by all thoughts # staggered: thought k starts ONE step behind k-1, # overlapping in time — but still SEES earlier thoughts for k in range(K): z[k] = shared for layer in range(Nr, L): # TOP layers only z[k] = layer(z[k], attend_to = states_of[:k]) # ← z[k] depends on z[<k] # no sample() — z[k] stays a continuous latent # depends on earlier thoughts' IN-PROGRESS states, # not their finished output → no full wait

The two lines that matter

Dependencystate = state + embed(tok) — thought k+1 needs thought k's finished token → must wait for it to fully completeattend_to = states_of[:k] — thought k still depends on earlier thoughts, but on their in-progress states (ready just-in-time thanks to the stagger) → no full wait

Output typesample(h) → a discrete word (fed back in)a continuous latent z (never turned into a discrete token)
Gradients flow?No — sampling a discrete token breaks the chainYes — continuous all the way → "gradient-continuous VLA path"
The punchline: next-token is slow because each thought waits for the previous one to fully finish (and re-climbs the whole network). Staircase keeps the same thought-to-thought dependency, but the stagger lets thought k read earlier thoughts' in-progress states instead of waiting for their finished output — so it overlaps in time and shares the bottom flight. Latency drops from K×L ticks to about L + (K−1). And because it emits continuous latents instead of sampling discrete words, training signals can flow straight from language reasoning into the robot's actions — which is exactly what a real-time World Action Model needs.
