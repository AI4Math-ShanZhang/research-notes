① Chunks & Events

What is a "chunk"?
A robot is driven by tiny commands, ~10–50 per second (move hand a few mm, move gripper). A chunk is a fixed-length block of these consecutive steps predicted all at once — e.g. the next 16 steps (~half a second) in one shot, instead of one-at-a-time.
One line: a chunk = a short pre-planned burst of movement.
Why use chunks at all?
  • Speed — one model call covers many steps, instead of querying per step.
  • Smoothness — committing to a burst avoids the jitter of re-deciding every step.
  • Less error pile-up — a coherent short plan drifts less than step-by-step choices.
Chunk vs. event — what's the difference?
A fixed chunk is cut by the clock (always the same length) — so a boundary can land mid-behavior and mix "reach" + "grasp" in one unit. An event is cut where the behavior changes (variable length), so each unit holds exactly one nameable behavior.
One line: chunk = equal slices that ignore what's happening; event = whatever length holds one coherent behavior. WALL-WM cuts at events.
How are chunks connected? Fed sequentially in training?
At run-time: a rolling loop — predict a chunk → execute → re-observe the new scene → predict the next. They connect through the real world, not as one long pre-planned stream.

In training: events are learned as mostly-independent units. Fixed-chunk (unified) mode adds a short history window so a chunk isn't ambiguous under one global instruction. Multiple events may be packed into one sequence for GPU efficiency, but an attention mask blocks them from seeing each other.
If a chunk mixes two behaviors, can the model still imagine/act?
Yes — generation still works. The video model can draw the reach→grasp transition, and the action model can output the blended motion. Mixing doesn't break it; it just costs you:
  • no clean label → muddy word→motion mapping (weaker generalization),
  • ambiguity about where you are in the task (needs a history window),
  • delicate contact moments cut through and pre-committed → fumbles.
Events are cleaner on all three.

② Staircase decoding

What is staircase decoding, in one picture?
A way to produce the reasoning "thoughts" fast. Instead of climbing the whole network once per thought (serial), it computes the bottom layers once (shared grounding) and staggers the top layers like a pipeline, so thoughts overlap in time.
One line: share the bottom flight, stagger the top — most of the reasoning of serial, most of the speed of parallel.
What does "reach the top" mean?
A network is a stack of layers; information climbs from the bottom (input) to the top (output). "Reaching the top" = finishing the forward pass = one thought is fully computed. Serial finishes them far apart; staircase finishes them in one wave.
Why not just parallelize all thoughts across all layers?
Two reasons:
  • the bottom layers do identical grounding for every thought — recomputing it K times is wasteful;
  • fully-independent parallel deletes the thought-to-thought dependency that reasoning needs (thought 2 could never see thought 1).
Staircase shares the bottom and staggers the top to keep the dependency while overlapping in time.
Does thought 2 depend on thought 1? (correcting the earlier code)
Yes it does — the earlier parallel_for … return z was misleading. z₂ depends on z₁, but on z₁'s in-progress intermediate states (ready just-in-time thanks to the one-step stagger), not on z₁'s finished output.
One line: same dependency as serial, minus the hard wait.
If nothing is sampled, how is the world generated?
The staircase makes a plan (continuous latents), not the world. The video model and action model generate the world/actions by denoising (diffusion / flow matching), guided by those latents. No sample() anywhere → the whole path stays continuous and trainable ("gradient-continuous").

③ Cross-view masking

Sight-cone masking — the idea?
Each image patch is a cone of vision from its camera. Two patches from different cameras may attend to each other only if their cones overlap in 3D (they could see the same spot). Impossible pairs get a −∞ attention bias → blocked. Training-only.
How is co-visibility computed? What parameters?
  1. Build each cone from camera calibration (position, orientation, lens).
  2. March along each axis in the depth band [d_min, d_max].
  3. Find the closest approach of the two axes.
  4. Cone radius grows with depth (r = t·tan γ).
  5. Co-visible if gap ≤ r₁ + r₂.
Tunable knobs: cone width γ·l and depth band [d_min, d_max] (both control leniency). Everything else is fixed by calibration.
What is "depth" here?
The unknown distance from a camera to the object along a patch's ray. A single RGB camera can't measure it, so co-visibility uses a plausible near–far band instead of the exact value. No depth sensor needed — just calibration + the band.
Tube patch masking — what and why?
Take a k×k window on one view, held across all frames (a "tube"), and replace it with noise. Now the model can't recover it from that camera's other frames (no temporal shortcut) — the only path left is to attend to the other cameras. This forces cross-view attention to be learned. The window is sampled from the co-visible set U so recovery is actually possible.
Pair: sight-cone says which cross-view links are allowed; tube creates a need to use them.
What is p_tube?
A hyperparameter: the probability that a training sample gets a tube mask. A dial for how often the model faces the cross-view challenge (with p_tube^cond = chance of also masking the conditioning y, k = hole size, λ_mask = how hard the tube is weighted).

④ World model & the conditioning y

What is y?
The conditioning channel — the known/observed current frames, fed inpainting-style to guide generation. It's the visual condition (what the scene looks like now), separate from the text condition (what task to do).
Is y the previous frame — like t₀ for predicting t₁?
Yes, exactly. y = the current observed frame(s) — one keyframe per camera (t₀) — and the model generates the following frames (t₁, t₂, …). In the rolling loop, after acting, the new observation becomes the next y. It's the anchor, given, never generated.
Why does y have the same shape as the noisy video tokens?
So they line up position-by-position. The timeline is [observed │ future │ future …]. y carries content only where something is known (t₀) and is blank/masked at the future slots; the noisy tokens are noise everywhere. Same shape = every generated token has a matching "what do I know here?" token.
What does "inpainting channel" mean?
Generating the future from the present is inpainting across time (fill the "hole" of the future using the known present). "Channel" = the known content is appended as extra channels onto each token — a sticky-note per position saying "here's what's observed here (or: nothing)." Blanking those channels in the tube = "can't peek at the answer."
What are the per-layer video features h^V, and why "× depth"?
h^V = the video DiT's hidden states — the partly-processed video at a given layer. Depth = the number of stacked layers, so there's one h^V set per layer ("× depth"). The action tower reads them at matching depths (layer-wise coupling), giving it video understanding at all levels of abstraction — not just the final output.

⑤ Actions & tokens

The proprio state s — is it just numbers?
Yes — a small vector read from the robot's own joint sensors: hand positions, orientations, gripper openings (e.g. [14] floats). It's the robot's sense of where its body is now — an input, measured — versus the action trajectory it outputs to say where to move next.
What is an "action token"? Is it an embedding like the video tokens?
Yes — a D-dimensional embedding, the action twin of a video token. [T_a, D] = T_a movement-vectors. An encoder lifts the raw action numbers [T_a, A] up into token space; the MLP head brings them back down. Both towers share the same width D so they can cross-attend.
Are the "noisy action tokens" the same as [MASK] tokens?
No. A [MASK] token is a discrete placeholder symbol (BERT/MaskGIT style). A noisy action token is a continuous vector = real action + noise (pure noise at the start), which the model denoises rather than "unmasks."
