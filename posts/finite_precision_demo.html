“Each position holds only about log N bits” — see it for yourself

Three sliders. By the end the sentence “a latent vector is really just a few tokens” should feel obvious.

1 · Why counting N things needs log₂ N bits

A “bit” is one yes/no switch. With b switches you can make 2b different patterns — so you can give 2b things a unique name. To name N positions you need enough switches that 2b ≥ N, i.e. b = log₂ N bits.

Bits needed to label every position: (⌈log₂ N⌉)

2 · “Finite precision” = a continuous number snaps to 2b buckets

A real number has infinite digits (0.63700241…). A computer keeps only b bits, so the line from 0 to 1 is chopped into 2b buckets and your value snaps to the nearest one. Fewer bits → coarser line.

3 · The punchline: a latent vector = a few tokens

A PLT (looped transformer) passes a “latent vector” of numbers. But each number is finite-precision — just b bits from Part 2. So the whole vector is a fixed pile of bits… which is exactly what a handful of discrete tokens is. Watch them round-trip with zero loss.

4 · Two different budgets: vocab sets token size, length N sets how big a number gets

People fuse these two, but they’re separate. Vocabulary V = how many bits fit in one token (fixed, never grows). Sequence length N = how many bits an internal number needs (a position index or a count over N positions). To store one latent number in tokens you spend log N / log V of them.

5 · Why the precision rides on the question size N, not the answer length

The one-line reason. The latent precision is set by how many positions the model has to point at and count over while it computes — and that’s the whole working canvas, whose size is poly(N). So: 
latent precision = log(number of positions it indexes over)
                 = log(canvas)
                 = log(poly(N))
                 = about log N      ← driven by the QUESTION size N

The answer length never enters, because the answer is just a slice of that canvas — the model still has to address positions across the entire canvas to compute it.

Why the canvas, not the answer

Think about what a number inside the model is actually doing during denoising:

  • It has to say “go read position 738” — addressing a cell in the canvas.
  • Attention has to sum / count over all the cells — an aggregate over the whole canvas.

Both are about the total number of cells the model operates on = the canvas = poly(N). The 6 answer cells don’t get their own smaller budget — they live inside the same 1000-cell canvas, and computing them means reading and writing all over that canvas.

Concrete numbers

Question N = 100 tokens
Canvas 1000 cells (poly(N), e.g. 10·N — answer + scratch)
Answer just 6 cells

address one of 1000 cells → log₂ 1000 ≈ 10 bits
count over up to 1000 cells → up to ~10 bits
⇒ each internal number carries ~10 bits ≈ log N

Notice the 6 (answer length) played no role. Even the 6 answer cells sit among the 1000, so the model’s addressing still spans the full canvas. The precision tracks the 1000, and the 1000 is poly(N) — driven by the question size.

The reassuring footnote. Even if you tried to tie precision to the answer length, you’d get the same answer: the answer is itself at most poly(N), so log(answer) is also O(log N). They collapse to the same order. But the honest driver is the one above — precision = log(positions I index over) = log(canvas) = log(poly(N)) = log N.
question size N → canvas sized poly(N) → model indexes over poly(N) positions → each number needs log N bits

The question size sets the scale of everything downstream — which is exactly why the latent precision rides on N, not on the answer length.

The one sentence to keep

Under finite precision a “latent vector” is never holding more than a small, fixed number of bits. A few tokens hold the same bits. That is why MDM ≡ PLT: same information, one written as vectors, the other as symbols. The word “latent” makes it sound like more — it isn’t.
