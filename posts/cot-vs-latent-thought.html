Chain of Thought vs Latent Thought

Paper: A Formal Comparison Between Chain of Thought and Latent Thought — Kevin Xu & Issei Sato, arXiv:2509.25239.

✅ Conclusion

  • The two results: latent thought enables efficient parallel computation; CoT enables randomized approximate counting.
  • Practical guidance: latent reasoning suits problems that can be solved efficiently (structured, parallelizable); CoT is more effective for more complex problems where you fall back on sampling/approximation.
  • Future work: (1) can distillation cut the number of loop iterations without losing computational power? (2) extend the analysis to diffusion language models, which have both parallelizability and stochasticity — i.e. potentially the strengths of both paradigms.

📄 Paper Summary

The paper asks: chain-of-thought (CoT) and latent thought (looped-Transformer style) both do "more computation = more reasoning" — which is stronger, and for what? The answer is a two-sided separation: each beats the other on a different class of problems.

All three reuse the same Transformer block in a loop; they differ only in what is fed back each step.

The two separation results

  • Latent thought → parallel power. A looped TF run for d iterations behaves like a circuit of depth d; with polylog iterations it captures TCk. CoT with the same step budget only reaches TCk−1, because it must serialize intermediate results into tokens — its steps scale with the size of the computation, not its depth.
  • CoT → approximate counting. CoT samples a real token each step, a genuine source of randomness, so it can run Monte-Carlo approximation (FPRAS) for approximate counting and sampling. Latent thought is a deterministic map, so it cannot. Both Coconut and looped TF are strictly weaker here.

🖊 My Key Notes

The whole story in one line: the argmax in CoT is an information bottleneck — a whole continuous vector crushed to one of a few tokens (~17 bits) per step. Everything follows from whether a paradigm pays that toll.

Coconut, in plain terms

Normal CoT thinks in words; Coconut lets the model think in vectors. It deletes the "pick a word" step and feeds the raw hidden vector straight back, so it can carry "60% A, 40% B" forward instead of collapsing to "A". It's the bridge: continuous like the loop, but grows in length like CoT.

Continuous CoT (Coconut) is powerful because a continuous thought vector can be a superposition — it holds many possible reasoning paths at once instead of committing to one. Concurrent work (Zhu et al., 2025) makes this precise: on directed graph reachability, a continuous thought encodes a parallel breadth-first search (all frontiers alive at once), solving it in D (graph diameter) steps, whereas discrete CoT must sample a single path per step and needs ~n² steps.

Claim A — parallel reasoning (tournament analogy)

8 players, find the champion.
  • CoT narrates every match one at a time → 7 statements (the number of matches = size).
  • Looped TF plays a whole round at once: 4, then 2, then 1 → 3 rounds (depth = log₂ 8).

What log(n) buys: scalability. A billion-item problem finishes in ~30 parallel rounds; doubling the problem adds just one round. On parallel hardware, rounds ≈ wait time — so huge problems still finish fast.

Proof sketch, in plain terms — scratchpad vs parallel layers

The paper's proof sketch is the same idea stated as a computation graph. Take a little pyramid — 3 nodes to compute, arranged in 2 levels of depth:

inputs:   a    b    c    d
level 1:    e=a+b     f=c+d
level 2:        g=e+f

What "scratchpad memory" means. CoT computes by writing intermediate results down as tokens, then reading them back — its token sequence is scratch paper. To compute g, attention looks back at the tokens it already wrote for e and f (retrieve predecessors), the feed-forward layer does the e+f, and the result is written as the next token. Powerful — it's how CoT does long multi-step work — but it emits one token per step.

Why the loop is parallel. The looped TF keeps every node as its own position and runs all their feed-forwards at once, so it finishes a whole level per iteration:

CoT (one token/step)Looped TF (one level/step)
Step 1write ecompute e and f together
Step 2write fcompute g
Step 3write g— done —
Total3 = # nodes (size)2 = # levels (depth)

Why CoT can't parallelize: a single forward pass reads all tokens in parallel, but its output is just one new token — so each step finishes only one node. The loop updates all positions' states every pass, finishing a whole level at once. Scale up a pyramid over 1024 inputs: depth ≈ 10, size ≈ 1023, so the loop takes ~10 steps and CoT ~1023.

The catch — "with sufficient size". Holding a 1024-node level at once needs ~1024 positions of width. So the loop spends width (space) to save steps (time); CoT spends steps but stays narrow. That trade is what "provided the model has sufficient resources" means.

Two kinds of "depth" — layers vs iterations

The "depth" in the proof is not the transformer's layer count. There are two separate axes:

  • Transformer depth = number of layers L. The fixed architectural stack (e.g. 12 layers). One forward pass goes through all L layers once. Same for all three methods.
  • Computation/graph depth D. How many sequential levels the problem needs (our pyramid: D=2).

For a looped TF, the relevant depth is the number of loop iterations T, not the layers. Loop an L-layer block T times → effective depth L×T of sequential computation, but only L layers' worth of weights (the block is reused). Each iteration handles one graph level, so you loop ≈ D times.

Layers (weights)Forward passesEffective depth
Plain TF, one passL=212 — only shallow problems
Looped TF, T iterationsL=2 (reused)TL×T — solves depth-(L·T)

Why it matters: one forward pass gives only L levels of sequential computation, so a plain transformer is stuck doing shallow work. Looping grows effective depth (L×T) cheaply — more computation, no more parameters. That extra effective depth is what maps to graph depth D.

Where CoT fits: same fixed depth L, but many decoding steps — each a fresh forward pass. Its sequential computation is carried through the scratchpad tokens across steps, not through layers or iterations. So CoT's reach is set by number of steps, which scales with size (nodes).

One line: transformer layers L = depth of one forward pass (fixed); latent thought's "depth" = loop iterations (effective depth L×T ≈ graph depth D); CoT's "depth" = decoding steps, carried by the scratchpad and scaling with size.

Claim B — approximate counting (marbles analogy)

A jar of a million marbles — how many are red? Counting exactly is hopeless; grab a random handful of 100, see 30 red → estimate ~300,000. Works only if the handful is truly random.

CoT's next-token sampling is that random handful. What randomness buys: feasibility. Some counts (2100 possibilities) can never be computed exactly, but sampling gets within a few percent in a fraction of a second. The benefit isn't "faster" — it's "possible at all."
