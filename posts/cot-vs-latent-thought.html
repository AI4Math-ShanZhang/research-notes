Chain of Thought vs Latent Thought — Key Points

Paper: A Formal Comparison Between Chain of Thought and Latent Thought — Kevin Xu & Issei Sato, arXiv:2509.25239. Companion demo: cot_loop_coconut_demo.py.

The whole story in one line: both paradigms reuse the same Transformer block in a loop — they differ only in what gets fed back each step, and that single difference gives each one a different superpower. Latent thought → parallel reasoning. CoT → approximate counting by sampling.

1. The three settings: one difference that matters

All three run the same block repeatedly. The only thing that changes is what crosses the loop boundary.

SettingWhat is fed back each stepDiscretized?Sequence grows?
CoT1 token id (argmax of the last hidden vector, then re-embedded)Yes — argmax bottleneckyes (+1 token)

Coconutthe last hidden vector, appended as a raw vectornoyes (+1 position)
Coconut in plain terms (Hao et al., Meta 2024): normal CoT thinks in words; Coconut lets the model think in vectors. It deletes the "pick a word" step — instead of decoding the hidden vector into a token and re-embedding it, it feeds the raw vector straight back. This keeps the full information (a vector can hold "60% A, 40% B" instead of collapsing to just "A"), so it can carry several possibilities forward at once. It is the bridge: continuous like the loop, but grows in length like CoT.
flowchart LR subgraph CoT A1[hidden vec] -->|argmax| A2[one token] A2 -->|re-embed| A3[next input] end subgraph Coconut B1[hidden vec] -->|append raw vector| B2[next input] end subgraph LoopedTF C1[full hidden matrix] -->|feed all back| C2[next input, same width] end

Why the argmax matters: CoT crushes a whole continuous vector into one of a few tokens each step (~17 bits for a real vocabulary). That is an information bottleneck. The two continuous methods skip it.

2. Claim A — Latent thought enables parallel reasoning

Tournament analogy. 8 players, find the champion.
  • CoT must narrate every match one at a time — "A beats B", "C beats D"… → 7 statements (the number of matches = the size).
  • Looped TF plays a whole round at once: 4 matches, then 2, then 1 → 3 rounds (the depth = log₂ 8).

The loop carries its full state, so it can finish a whole layer of work in one step. CoT can only utter one token per step, so it is stuck doing items one by one. Result: for wide, structured problems (adding many numbers, evaluating a big formula, graph connectivity), the loop needs ≈ log(n) rounds while CoT needs ≈ n steps. In the paper's terms, the loop reaches TCk; CoT with the same step budget lands one level weaker (TCk−1).

A billion-item problem finishes in 30 rounds. Doubling the problem adds just one round. On parallel hardware, rounds ≈ actual wait time — so huge problems still finish fast.

3. Claim B — CoT enables approximate counting

Marbles analogy. A jar with a million marbles — how many are red? Counting exactly means dumping all million out (hopeless). Instead grab a random handful of 100, see 30 are red → estimate ~30% → ~300,000. Fast and close — but only if the handful is truly random.

Where do the dice come from? When CoT picks its next word it samples from a probability distribution — a weighted dice roll. So CoT naturally does random sampling and can run "grab a random sample and estimate" methods (Monte Carlo / approximate counting). Latent thought is deterministic — same input always gives the same output, no dice — so it cannot do this trick. Both Coconut and looped TF are strictly weaker than CoT for approximate counting/sampling.

What randomness buys you: feasibility

Some counts are genuinely impossible to compute exactly — 2100 possibilities would take longer than the age of the universe. Random sampling gives an answer within a few percent in a fraction of a second, and you can dial accuracy up with more samples. The benefit isn't "a bit faster" — it's "possible at all." For a whole class of problems, sampling is the only known way to get an answer in reasonable time.

4. The takeaway

ParadigmSuperpowerBenefitGood at

Neither dominates. One is a team working in parallel; the other is a pollster who takes random samples. Reach for looped/latent when the bottleneck is deep parallel structure; reach for CoT when the problem is hard to solve exactly but approximable by sampling.
